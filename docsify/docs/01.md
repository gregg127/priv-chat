# Private Chat Application – Architecture Specification

## Specification-Driven Development (SDD) Entry Document

---

# 1. Overview

This document defines the architecture, components, protocols, and security model for a private, real-time chat system using the Signal Protocol for end-to-end encryption (E2EE).

## Goals

- Private messaging with strong cryptographic guarantees
- Multi-user chat rooms
- Global room discovery
- Password-based access control
- Specification suitable for AI-driven implementation

---

# 2. System Requirements

## Functional Requirements

1. Users authenticate using a password
2. Users can view a global list of rooms
3. Users can join rooms
4. Rooms support multiple participants
5. Messages are end-to-end encrypted
6. Server cannot read message contents

## Non-Functional Requirements

- Low latency (<200ms message delivery target)
- Horizontal scalability
- Cryptographic security (forward secrecy, deniability)
- Stateless or minimally stateful backend

---

# 3. High-Level Architecture

## Components

### 3.1 Client Application

- Web-based (initial implementation)
- Responsibilities:
  - Key generation and management
  - Encryption/decryption
  - Communication with server
  - UI/UX

### 3.2 Backend Server

- Responsibilities:
  - Authentication
  - Room management
  - Message relay (no decryption)
  - Key distribution (public keys only)

### 3.3 Storage

- Stores:
  - User credentials (hashed)
  - Public identity keys
  - Room metadata

---

# 4. Communication Model

## Protocol

- WebSocket (primary real-time transport)

## Events

### Client → Server

- AUTHENTICATE
- JOIN_ROOM
- LEAVE_ROOM
- SEND_MESSAGE
- FETCH_ROOMS
- REGISTER_KEYS

### Server → Client

- AUTH_SUCCESS / AUTH_FAILURE
- ROOM_LIST
- USER_JOINED
- USER_LEFT
- RECEIVE_MESSAGE
- KEY_BUNDLE

---

# 5. Authentication

## Method

- Password-based authentication

## Flow

1. Client sends password
2. Server validates against stored hash (bcrypt)
3. Server returns session token (JWT)

---

# 6. Room System

## Structure

```
Room {
  id: string
  name: string
  participants: list<UserID>
}
```

## Properties

- Global visibility
- No encryption at server level
- Encryption handled client-side

---

# 7. Signal Protocol Integration

## Key Types

### Identity Key Pair

- Long-term key
- Generated once per user

### Signed PreKey

- Medium-term key
- Signed by identity key

### One-Time PreKeys

- Ephemeral keys for session initiation

---

## Key Storage

### Server Stores:

- Public identity key
- Signed prekey
- One-time prekeys

### Client Stores:

- Private keys (secure storage)
- Session state

---

# 8. Session Establishment

## X3DH (Extended Triple Diffie-Hellman)

### Steps:

1. Sender fetches recipient key bundle
2. Sender computes shared secret using:
   - Identity key
   - Signed prekey
   - One-time prekey
3. Sender sends initial encrypted message
4. Receiver derives same shared secret

---

# 9. Message Encryption

## Algorithm

- Double Ratchet Algorithm

## Properties

- Forward secrecy
- Break-in recovery
- Message unlinkability

---

## Message Format

```
EncryptedMessage {
  header: {
    ratchetKey: bytes
    counter: number
  }
  ciphertext: bytes
}
```

---

# 10. Group Messaging

## Approach

- Pairwise encryption per participant

## Process

1. Sender encrypts message separately for each participant
2. Server broadcasts encrypted payloads
3. Clients decrypt individually

---

# 11. Security Model

## Guarantees

- End-to-end encryption
- Server cannot decrypt messages
- Forward secrecy
- Authentication via identity keys

## Threat Model

- Malicious server
- Network interception
- Replay attacks
